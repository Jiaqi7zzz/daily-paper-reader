## 摘要
编程智能体在大型代码库中执行任务的前提是代码定位——即识别出需要处理的相关文件、类和函数。虽然代码库级别的代码定位通常使用基于嵌入的检索方法（如向量搜索）来完成，但最近的研究重点在于开发能够定位相关代码的智能体，这种定位既可以作为独立的前置步骤，也可以与实际工作交替进行。大多数先前的智能体代码搜索方法为智能体配备了复杂的专用工具，例如通过静态分析生成的代码库图。在本文中，我们证明了通过有效的强化学习方案，仅配备标准 Unix 终端的编程智能体也能通过训练获得出色的结果。我们在三个基准测试（SWE-Bench Verified、Pro 和 Lite）上的实验表明，我们的模型始终优于或等同于比其大 2-18 倍的基础及经过后训练的大语言模型（LLM），且在某些情况下，即使后者使用了专门的脚手架（scaffolds），我们的模型也能接近 Claude Sonnet 等闭源模型的性能。我们的工作特别关注于将现有的编程智能体环境重新用于代码搜索的技术、奖励设计以及强化学习优化。我们发布了由此产生的模型系列 CodeScout，以及所有代码和数据，供社区在此基础上进一步开发。

## Abstract
A prerequisite for coding agents to perform tasks on large repositories is code localization - the identification of relevant files, classes, and functions to work on. While repository-level code localization has been performed using embedding-based retrieval approaches such as vector search, recent work has focused on developing agents to localize relevant code either as a standalone precursor to or interleaved with performing actual work. Most prior methods on agentic code search equip the agent with complex, specialized tools, such as repository graphs derived from static analysis. In this paper, we demonstrate that, with an effective reinforcement learning recipe, a coding agent equipped with nothing more than a standard Unix terminal can be trained to achieve strong results. Our experiments on three benchmarks (SWE-Bench Verified, Pro, and Lite) reveal that our models consistently achieve superior or competitive performance over 2-18x larger base and post-trained LLMs and sometimes approach performance provided by closed models like Claude Sonnet, even when using specialized scaffolds. Our work particularly focuses on techniques for re-purposing existing coding agent environments for code search, reward design, and RL optimization. We release the resulting model family, CodeScout, along with all our code and data for the community to build upon.

*   **核心问题**：在处理大规模代码库任务（如 SWE-Bench）时，智能体面临的首要挑战是**代码定位（Code Localization）**，即从成千上万的文件中准确识别出需要修改的文件、类和函数。
*   **研究动机**：
    *   **现有方法过于复杂**：目前的智能体代码搜索通常依赖特定编程语言的静态分析工具（如 AST 解析、代码依赖图），这增加了部署难度且难以跨语言通用。
    *   **效率与成本**：依赖闭源大模型进行定位成本高昂，且通用模型在复杂仓库导航中容易产生“上下文腐烂”或定位错误。
    *   **研究空白**：虽然工业界传闻强化学习（RL）能提升定位能力，但缺乏公开、可复现的训练方案和开源模型。

#### 2. 方法论：核心思想与关键技术
*   **核心思想**：证明通过精细设计的强化学习方案，一个仅配备**标准 Unix 终端（Bash）**的智能体，无需任何专用工具，也能实现顶尖的代码定位性能。
*   **关键技术细节**：
    *   **OpenHands-Bash 脚手架**：智能体仅使用 `rg` (ripgrep)、`grep`、`ls`、`sed` 等标准命令进行仓库探索。
    *   **奖励设计（Reward Design）**：采用多粒度 F1 分数作为奖励信号。公式为：$R = F1_{file} + F1_{module} + F1_{func}$。针对 14B 模型，还加入了步数惩罚奖励以鼓励在限制步数内完成任务。
    *   **训练算法**：使用 **GSPO（Group Sequence Policy Optimization）**，这是一种改进的策略梯度算法，通过序列级重要性采样和紧凑的剪切（Clipping）来稳定训练。
    *   **训练流程**：
        *   **4B/14B 模型**：直接从基础模型进行 RL 训练。
        *   **1.7B 模型**：先利用 14B 模型的成功轨迹进行拒绝采样微调（RFT），再进行 RL 训练。
    *   **数据构建**：从 SWE-Smith 提取 39K 个实例，通过解析 Ground Truth 补丁自动生成文件、模块、函数三级的定位标签。

*   **实验规模**：涵盖了从 1.7B 到 14B 的三个模型尺寸，并在三个主流基准上进行了全面测试。

#### 6. 主要结论与发现
*   **小模型逆袭**：CodeScout-1.7B 的定位性能超过了比其大 8 倍的 Qwen3-14B 基础模型；CodeScout-14B 在函数级定位上甚至超越了使用复杂工具的 Claude-3.7-Sonnet。

*   **方案简洁通用**：基于 Bash 终端的方案天然具备跨语言潜力（虽然目前主要测试 Python）。
*   **全栈开源**：公开发布了模型权重、训练代码、环境和数据集，极大地推动了社区对编程智能体 RL 训练的研究。
*   **训练效率高**：通过 RFT 预热和 GSPO 算法，实现了在有限算力下训练出高性能的小型专业化模型。

*   **语言覆盖有限**：目前主要针对 Python 仓库进行训练和评估，尚未在 Java、C++ 等其他语言上验证其泛化性。
